| 1 | RAG là gì | RAG (Retrieval-Augmented Generation) là một kỹ thuật tiên tiến giúp nâng cao khả năng truy xuất và sinh văn bản của các Mô hình ngôn ngữ lớn (LLM) bằng cách kết hợp chúng với các nguồn tài liệu tri thức bên ngoài. | | |
| 2 | Vì sao chọn model sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2 cho embedding | Dự án Xiaozhi Robot chọn model `paraphrase-multilingual-MiniLM-L12-v2` cho embedding vì những lý do chính sau đây:  *   **Khả năng hỗ trợ đa ngôn ngữ:** Model này hỗ trợ hơn 50 ngôn ngữ, bao gồm cả tiếng Việt. Điều này giải quyết vấn đề truy vấn xuyên ngôn ngữ (cross-lingual retrieval) mà model trước đó (`all-MiniLM-L6-v2`) gặp phải, vốn chủ yếu được huấn luyện trên tiếng Anh. *   **Cải thiện độ chính xác:** Sau khi chuyển sang model này, khả năng khớp ngữ nghĩa được cải thiện rõ rệt. Các câu hỏi trước đó bị "miss" hoàn toàn đã có thể nằm trong top 3 đến top 5 kết quả tìm kiếm. *   **Hiệu năng phù hợp:** Model này vẫn thuộc họ MiniLM nên đảm bảo tính nhẹ, tốc độ nhanh và chạy tốt trên CPU, phù hợp với yêu cầu của dự án. | | |

Tuy nhiên, tài liệu không mô tả chi tiết định dạng dữ liệu cụ thể (cấu trúc JSON hoặc các trường dữ liệu) mà `search_knowledge` trả về. | 6 | Thiết bị gửi tin nhắn hello khi mở kết nối WebSocket như thế nào | Chào bạn, với tư cách là trợ lý AI cho dự án Xiaozhi Robot, tôi xin giải đáp câu hỏi của bạn dựa trên tài liệu kỹ thuật như sau:  Ngay sau khi kết nối WebSocket được thiết lập, thiết bị sẽ gửi một tin nhắn JSON có dạng `{"type": "hello"}` để thông báo các tham số của thiết bị tới server.  Cấu trúc chi tiết của tin nhắn này bao gồm: *   **`type`**: Phải là `"hello"`. *   **`version`**: Phiên bản giao thức (ví dụ: `1`). *   **`transport`**: Phương thức vận chuyển, ở đây là `"websocket"`. *   **`features`** (không bắt buộc): Chứa các tính năng được cấu hình khi biên dịch, ví dụ: `"mcp": true` (hỗ trợ MCP) hoặc `"aec": true` (sử dụng AEC từ phía server). *   **`audio_params`**: Các tham số âm thanh bao gồm:     *   `format`: Định dạng âm thanh (ví dụ: `"opus"`).     *   `sample_rate`: Tần số lấy mẫu (ví dụ: `16000`).     *   `channels`: Số kênh (ví dụ: `1`).     *   `frame_duration`: Thời lượng khung âm thanh (thường là `60` ms).  **Ví dụ về tin nhắn hello từ thiết bị:** ```json {   "type": "hello",   "version": 1,   "features": {     "mcp": true,     "aec": true   },   "transport": "websocket",   "audio_params": {     "format": "opus",     "sample_rate": 16000,     "channels": 1,     "frame_duration": 60   } } ```  Việc gửi tin nhắn này là bước khởi đầu để thiết bị và server xác nhận cấu hình trước khi bắt đầu các phiên trao đổi dữ liệu tiếp theo. | | |
| 7 | Tại sao không nên ghi đè cấu hình board có sẵn | Bạn không nên ghi đè cấu hình của board có sẵn vì các lý do sau:  *   **Rủi ro với bản cập nhật OTA:** Việc ghi đè cấu hình là rất nguy hiểm vì các bản cập nhật OTA (Over-The-Air) có thể thay thế firmware tùy chỉnh của bạn bằng firmware gốc cho board ban đầu đó. *   **Tính định danh và kênh cập nhật:** Mỗi board cần có một danh tính duy nhất và kênh cập nhật firmware riêng biệt. Nếu bạn cần tùy chỉnh IO khác với board có sẵn, hãy luôn tạo một loại board mới hoặc sử dụng mảng `builds` trong `config.json` để tạo ra một tên firmware riêng biệt với các macro `sdkconfig` khác nhau. | | |
| 8 | Partition v2 có gì khác so với v1 | Dựa trên ngữ cảnh được cung cấp, dưới đây là những khác biệt chính giữa Partition Table v2 so với v1:  **1. Khả năng tương thích:** *   Phiên bản v2 không tương thích với v1, do đó không thể nâng cấp từ v1 lên v2 thông qua OTA. Để chuyển sang v2, bạn cần phải flash firmware thủ công.  **2. Cải tiến cấu trúc phân vùng (Partition Layout):** *   **Thêm phân vùng `assets`:** Phiên bản v2 bổ sung phân vùng `assets` để hỗ trợ nội dung tải qua mạng, thay thế cho phân vùng `model` cũ (960KB) của v1. *   **Thay đổi dung lượng phân vùng ứng dụng:** Các phân vùng ứng dụng (`ota_0`, `ota_1`) trong v2 đã được giảm kích thước (xuống còn 4MB mỗi phân vùng) để nhường không gian cho phân vùng `assets`.  **3. Đặc điểm phân vùng `assets`:** *   Trong v2, phân vùng `assets` cho phép lưu trữ và cập nhật nội dung linh hoạt qua mạng (HTTP) mà không cần flash lại thiết bị, bao gồm:     *   Các mô hình từ đánh thức (wake word models).     *   Hệ thống giao diện (Theme): font chữ, hiệu ứng âm thanh, hình ảnh nền, yếu tố UI, emoji và file cấu hình ngôn ngữ.  **4. Sự linh hoạt theo dung lượng Flash:** *   v2 tối ưu hóa bố cục phân vùng cho các kích thước flash khác nhau:     *   Thiết bị ESP32-C3 có phân vùng `assets` nhỏ hơn (4MB) do giới hạn trang mmap.     *   Thiết bị 32MB có phân vùng `assets` lớn nhất (16MB).  **Tóm tắt so sánh cấu trúc (trên thiết bị 16MB):** *   **v1:** `model` (960KB), `ota_0` (6MB), `ota_1` (6MB). *   **v2:** `ota_0` (4MB), `ota_1` (4MB), `assets` (8MB). | | |
